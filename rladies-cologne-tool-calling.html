Introduction to LLMs and tool calling

Is it magic, or not so magic after all?

What are LLMs?

  • LLM = Large Language Model
  • Trained on huge amounts of text data
  • Designed to predict and generate human-like language


All powered by a model that recognises patterns at scale.

How do LLMs work?

The answer is quite boring: statistics.

The magic of scale

And something cool happens when you train a model on massive datasets with billions of words: LLMs start to show emergent abilities, aka skills the model wasn‚Äôt directly trained for, yet still manages to do.


An LLM was trained to predict the next word in a sentence, but at scale it can also:

  • Write code
  • Translate languages
  • Do fun things like writing poetry or jokes

What are LLMs good at?

LLMs are great at:

  • Generating human-like text
  • Providing conversational responses
  • Summarising text
  • Answering questions based on provided context
  • Writing and debugging code snippets

What are LLMs bad at?

It isn‚Äôt all rainbows and unicorns though.


LLMs can be confidently wrong. They can ‚Äúhallucinate‚Äù information, making up facts or details that aren‚Äôt true:

  • Counting
  • Performing calculations
  • Providing real-time or up-to-date information

Why do LLMs hallucinate?

LLMs generate text based on patterns in the data they were trained on.


They don‚Äôt have a true understanding of the world or access to real-time information.


So, when asked a question outside their training data, they might produce plausible-sounding but incorrect answers.

LLMs don‚Äôt execute code

LLMs can generate code snippets, but they don‚Äôt actually run or test the code.

  • If we ask an LLM to write a function, it might produce code that looks correct but isn‚Äôt.
  • If we ask an LLM to perform calculations, it might give us the wrong answer because it‚Äôs generating text based on patterns rather than executing mathematical operations.
  • If we ask for real-time data, like the current weather, it can‚Äôt provide accurate information because it doesn‚Äôt have access to live data.

But we know how to be accurate

What if we could combine the strengths of LLMs with tools that provide accurate information? Real functions that get executed?


Hello, tool calling!

What is tool calling?

Tool calling is a technique where an LLM can invoke external tools or functions to perform specific tasks.


For example: a tool that fetches real-time weather data. When the LLM is asked about the weather, it can call this tool to get accurate information instead of guessing.


Or, we can create a tool that performs calculations. When the LLM is asked a math question, it can call this tool to get the correct answer.

Benefits of tool calling

  • Accuracy: tools provide reliable and up-to-date information
  • Specialisation: tools can be designed for specific tasks
  • Efficiency: LLMs can focus on generating text while tools handle other stuff
  • Reduced hallucination: by relying on tools for factual information, the chances of LLMs making things up decrease

About the chat object

  • It‚Äôs stateful: it keeps track of the conversation history.
  • You give it a "provider/model" string when you create it, e.g. "claude/claude-sonnet-4-5".
  • You can also give it a system prompt to set the behaviour of the LLM.
  • Optionally you can pass it params to tweak model settings (we won‚Äôt get into that today).

System prompt vs user prompt

  • The system prompt is where you set the ground rules: it tells the model who it is, how it should behave, and what the answer should look like. It‚Äôs stable context.
  • The user prompt is the specific request you make right now, like asking a question or giving a command. It‚Äôs dynamic context.

What if we want to make the baby age dynamic, so we can use the same prompt for different ages?

Some tips

  • Prompt design is a real thing: the way you phrase your system and user prompts matters.
  • Provide enough context for the LLM to understand what you want. The more specific, the better (generally).
  • Be specific in your system prompt about the format you want the answer in.
  • Format your prompts (e.g. markdown) to make it more readable (for you and the LLM).
  • Make sure all that work doesn‚Äôt get lost: store your prompt in a separate file under version control.

Back to the output format

We asked the LLM to provide a list of activities with specific fields: fun score, duration, description.


We could even make it more specific by stating we want JSON, YAML, or a Markdown table.

It works‚Ä¶ until it doesn‚Äôt

This works, but is not always reliable. The LLM might forget to include a field, or format it incorrectly (e.g. wrap output in a code block, missing comma etc). The more complex the structure, the more likely it is to mess up. The solution?


Structured output!

Structured output

Structured output refers to LLMs that support structured schemas as part of the API call and not just as instructions in the prompt.


These tell the model not just what to output, but how to constrain the generation itself.


The model generates a structure directly (like a JSON object or typed dictionary), and the output is guaranteed to match that structure.

Requirements for structured output

To return structured output, we need to change two things:

  • Use chat_structured
  • Provide a type specification

Specification

Describe the structure of the output you want:

data_model <- type_object(
  baby_age_months = type_integer(
    description = "The age of the baby in months."
  ),
  activities = type_array(
    description = "List of 10 suggested activities for the baby based on the weather and age.",
    type_object(
      name = type_string(description = "Short title for the activity."),
      fun_score = type_integer(
        description = "Score from 1‚Äì10 indicating how fun the activity is likely to be."

Next up: tool calling!

Ok nice, but not tool calling yet.


Let‚Äôs say we want to suggest more specific activities, like going to the park, but only if the weather is nice. Or going to see the Eifel tower, but only if you‚Äôre in Paris.


We need to tell the model in which city we are, and what the current weather is.

Getting real-time data

Remember, LLMs can‚Äôt access real-time data on their own. We need to create tools to fetch this information for us. In ellmer, we can define functions that the LLM can call. For example:

#' Get the current weather for a given city using Open-Meteo API
#'
#' @param city The name of the city
#' @return A tibble with city, temperature in Celsius, windspeed, and weather code
get_current_weather <- function(city) {
  # Geocoding API to get latitude/longitude from city name
  geo_resp <- request("https://geocoding-api.open-meteo.com/v1/search") %>%
    req_url_query(name = city, count = 1) %>%
    req_perform() %>%
    resp_body_json()

  if (is.null(geo_resp$results)) {
    stop("City not found: ", city)
  }

  lat <- geo_resp$results[[1]]$latitude
  lon <- geo_resp$results[[1]]$longitude

  # Weather API for current weather at that location
  weather_resp <- request("https://api.open-meteo.com/v1/forecast") %>%
    req_url_query(latitude = lat, longitude = lon, current_weather = "true") %>%
    req_perform() %>%
    resp_body_json()

  current <- weather_resp$current_weather

  return(
    tibble(
      city = city,
      temperature_c = current$temperature,
      windspeed = current$windspeed,
      weather_code = current$weathercode
    )
  )
}

From function to tool

To make this function available as a tool for the LLM, we need to wrap it using tool():

get_current_weather <- tool(
  get_current_weather,
  "Returns the current weather for a given city",
  city = type_string(
    "The name of the city for which to get the weather",
    required = TRUE
  )

Register tool

‚Ä¶ and register it:

chat$register_tool(get_current_weather)

But‚Ä¶


‚ÄúNote that structured data extraction automatically disables tool calling‚Ä¶‚Äù


So how do we combine both?

Two step process

  1. First, we call the tool to get the current weather.
  2. Then, we pass that weather information to the structured chat function to get the baby activities.


Since our system prompt nicely lists these two tasks, we can let our user prompts be as simple as:

  • chat$chat("Execute Task 1 (get weather)")
  • chat$chat_structured("Execute Task 2 (get baby activities)", type = data_model)
